# 구조화 스키마 수작업 Annotation 가이드

> **한 줄 정의**
> 요소 크롭의 값을 **하나의 통짜 텍스트**(`"Ø15H7"`)가 아니라 **의미 단위로 쪼갠 named 필드**
> (`{nominalValue:"Ø15", toleranceClass:"H7"}`)로 라벨링하고,
> 그 쪼개기를 **정규식 자동**이 아니라 **사람이 직접/검수**하는 작업.

---

## 1. 왜 이 이야기가 나왔나 (배경)

Donut 값 인식 실험에서 모델이 읽은 결과를 **"정답과 완전히 똑같은가(exact)"** 로만 채점했더니
**정확도 0%** 가 나왔다. 사용 불가처럼 보인다.

하지만 글자 단위로 비교(**char 유사도**)해 보면 **약 0.47** — 즉 모델은 실제로 **절반쯤은 읽고 있었다.**
완전히 못 읽은 게 아니라, **"거의 맞았지만 한두 글자 틀린" 답까지 0점 처리**되어 진전이 가려진 것이다.

예시로 보면 명확하다:

| 정답 | 모델이 읽은 값 | exact 채점 | 실제 상태 |
|---|---|---|---|
| `Ø6.5` | `Ø6.4` | ❌ 0점 | 거의 맞음 (한 글자 차이) |
| `90±0.1` | `90` | ❌ 0점 | 핵심값은 맞고 공차만 놓침 |
| `25` | `5` | ❌ 0점 | 앞자리 누락 |

**요점**: "전부 맞아야 1점, 아니면 0점"인 채점은 **부분 정답을 못 본다.**
값을 의미 단위 필드로 쪼개 채점·학습하면 이 **부분 진전을 살릴 수 있다** — 그래서 구조화 스키마가 필요하다.

> 참고(실측): exact 정확도 `0.000` vs char 유사도 `0.468`. 같은 모델인데 채점 방식 때문에 평가가 크게 갈렸다.

이 간극을 줄이고, 더 깨끗한 정답으로 학습시키기 위한 방법이 **구조화 스키마 수작업 annotation**이다.

---

## 2. "구조화 스키마"란

값 문자열을 **카테고리별 필드 dict**로 분해한 형태다. (논문 Numerical VLM 방식)

```
"8X Ø6.5±0.1"  →  { "quantity":     "8",
                    "nominalValue": "Ø6.5",
                    "upperLimit":   "+0.1",
                    "lowerLimit":   "-0.1" }
```

클래스별 필드 정의(현재 `parse_to_schema` 기준):

| 클래스 | 필드 |
|---|---|
| Dimension | `quantity`, `nominalValue`, `upperLimit`, `lowerLimit` |
| GD&T_FCF | `geometricCharacteristic`, `tolerance`, `datumReference[]` |
| Surface_Roughness | `roughness` |
| Datum | `datum` |
| Hole_Callout | (스키마 없음 → 통짜 `value`) ⚠️ |

---

## 3. 현재 방식(자동) vs 제안(수작업)

| 구분 | 현재: **자동 분해** | 제안: **수작업 구조화** |
|---|---|---|
| 라벨 원천 | CVAT `value` 텍스트 **1칸** (`"8X Ø6.5±0.1"`) | CVAT에서 **필드별로 입력** (`quantity=8, nominal=Ø6.5, ±=0.1`) |
| 분해 주체 | 학습 시 `parse_to_schema` **정규식**이 추측 | **사람**이 분해(또는 자동분해를 검수·수정) |
| 정확성 | 규칙 못 맞으면 **조용히 오라벨** | 사람이 보장 |
| 결과 | "대충 맞는" 노이즈 정답 | 검증된 정답 |

핵심: 지금도 구조화는 **하고 있다**. 다만 **정규식이 추측**한다. 제안은 그 추측을 **사람이 책임지게** 하는 것.

---

## 4. 자동 분해가 실패하는 실제 사례

실제 라벨에 `parse_to_schema`를 돌려본 결과 — 대부분 맞지만 다음이 샌다:

| 원본 값 | 자동분해 결과 | 문제 |
|---|---|---|
| `Ø15H7` | `{nominalValue:"Ø15H7"}` | **H7(끼워맞춤 공차등급)이 분리 안 됨** — 정답은 `nominalValue:Ø15` + `toleranceClass:H7` |
| `M6X1.0 ⏥ .50` (Hole) | `{value:"M6X1.0 ⏥ .50"}` | **스키마 없어 통째로** — `thread:M6X1.0` + `counterbore:.50` 분리 안 됨 |
| `- 'A'±0.001 -` | `{upper:+0.001, lower:-0.001, nominalValue:"- 'A' -"}` | **쓰레기 입력**인데 규칙이 억지로 구조 생성 → 사람이면 걸러냄 |
| `.03 x 45°` | `{nominalValue:".03 x 45°"}` | `x`가 "수량 곱"인지 "모따기 by"인지 **모호** (`8X…`는 수량으로 처리되어 케이스마다 규칙이 다르게 동작) |

→ 이 노이즈가 **그대로 학습 정답**으로 들어가 모델 품질을 깎는다.

---

## 5. 수작업 구조화가 주는 이점

1. **부분 점수(partial credit) 가능**
   평가가 field-level P/R/F1이라, `nominalValue`만 맞고 `tolerance` 틀려도 **0.5점**.
   현재 exact F1 0.000에 묻히던 "절반의 진전"을 점수로 드러내 **학습 신호·평가가 매끄러워짐**.

2. **출력 안정화(output stability)**
   정답이 항상 같은 필드 구조 → 모델이 배우기 쉽고 hallucination↓.
   *(논문: "schema-constrained learning이 출력 변동성을 줄였다.")*

3. **라벨 정확성**
   `Ø15H7`·Hole·쓰레기값 같은 자동분해 실패를 사람이 바로잡음. **라벨 품질 = 모델 품질.**

4. **다운스트림 즉시 활용**
   `nominalValue=15, tolerance=H7`로 나오면 CAD/검사 DB가 바로 치수공차 검증 가능.
   통짜 `"Ø15H7"`은 또 파싱해야 함.

---

## 6. 비용 / 워크플로

- **라벨링 시간↑**: 크롭당 텍스트 1칸 → 필드 여러 칸.
- **CVAT 지원**: 박스 하나에 **여러 text attribute**(예: `quantity`, `nominal`, `upper`, `lower`) 부여 가능.
- **스키마 설계 선행**: 클래스별 필드 정의 확정 — 특히 지금 통짜인 **Hole_Callout 스키마 신설** 필요.
- **현실적 절충**: 자동분해를 **1차로 돌린 뒤 사람이 검수·수정**(`pending` 1,974건 검수 단계에 통합)하면
  처음부터 손으로 하는 것보다 빠르다.

---

## 7. 데이터 규모와의 관계 ⚠️ 중요

구조화 수작업 annotation은 **데이터 부족을 대체하지 못한다 — 보완재다.**

- 0% 벽의 1차 원인은 여전히 **"~1,800장으로 OCR을 처음부터 학습"**.
- 수작업 구조화는 (a) 적은 데이터에서 **신호를 더 짜내고** (b) **평가를 등급화**하지만,
  사용가능 수준(exact > 0)에 닿으려면 **데이터 증량(합성 + 추가 라벨)** 과 **함께** 가야 한다.

---

## 8. 두 핵심 레버와 로드맵

성능을 "사용 불가(0%)"에서 "논문 수준(F1 0.9대)"으로 끌어올리는 핵심 레버는 **두 가지**다.
**①은 "양", ②는 "질"** — 둘은 경쟁 관계가 아니라 **함께 곱해져야** 효과가 난다.

| # | 레버 | 무엇을 하나 | 왜 중요한가 | 비용 |
|---|---|---|---|---|
| **①** | **실 라벨 데이터 확충** (목표 ≥ **1만**) | CVAT로 element 크롭 + 값 라벨을 **대량 확보** | 논문이 F1 **0.935(0.963)** 를 낸 **결정적 레버**. 현재 ~1,800장(논문의 **1/7**)이라 OCR을 처음부터 못 배움 | **높음** (라벨링) |
| **②** | **수작업 구조화 스키마 annotation** | 값을 **의미 필드로 분해**해 라벨 | **부분점수 + 출력 안정**(논문 핵심) + 현재 **자동파싱 노이즈 제거** | 중~높음 |

### 왜 둘 다 필요한가 (양 × 질)

- **①만** 하면: 데이터는 많아지지만 라벨이 통짜/노이즈면 학습이 흔들린다.
- **②만** 하면: 라벨은 깨끗해도 표본이 적어(~1,800장) OCR 자체를 못 배운다 — 실험에서 3클래스로 줄여도 **0% → 0%** 였다.
- **①(양) × ②(질)** 가 곱해져야 논문 수준에 닿는다.

### 왜 이 순서인가 (②로 틀을 잡고 ①로 채운다)

스키마(②의 필드 정의·분해 규칙)를 **먼저 동결**한 뒤 대량 라벨링(①)을 해야 한다.
스키마 없이 1만 장을 통짜로 모으면 나중에 **전부 재분해**해야 하기 때문이다.

### 단계별 로드맵 (Phase 0 → 3)

**전체 흐름 한눈에** — 각 단계는 **앞 단계 산출물이 있어야 시작**된다(완료기준 = 다음 단계 진입 조건).

```
 Phase 0          Phase 1 ★결정적       Phase 2          Phase 3
 스키마 확정  ──▶  데이터 확충 ≥1만  ──▶  구조화 정합  ──▶  재학습 · 평가
 (레버 ②)         (레버 ①)            (레버 ②)         (확인)
 "틀 만들기"       "양 채우기"          "질 다듬기"       "성과 확인"
```

> 흐름의 논리: **②로 틀 → ①로 양 → ②로 질 → 평가.**
> ①(데이터 확충)이 **결정적 레버**이고, ②(구조화 스키마)가 그 **앞뒤에서 틀과 질을 받친다.**

각 단계를 **목표 / 작업 / 산출물 / 완료기준**으로 풀면:

---

#### ▷ Phase 0 — 스키마 확정  · 레버 ② · *틀 만들기*

| | |
|---|---|
| 🎯 **목표** | 라벨 구조를 **먼저 고정**해 이후 재작업을 없앤다 |
| 🔨 **작업** | 클래스별 필드 정의 동결(**Hole_Callout 스키마 신설**, `toleranceClass` 등 누락 보강) + CVAT 라벨에 **필드 속성 칸** 구성 |
| 📦 **산출물** | 확정된 필드 스키마 + CVAT 라벨 설정 |
| ✅ **완료기준** | 6개 클래스 전부 필드 정의·CVAT 입력칸 준비 완료 |

#### ▷ Phase 1 — 데이터 확충  · 레버 ① · *양 채우기*  ★**결정적**

| | |
|---|---|
| 🎯 **목표** | 실 라벨 크롭 **≥ 10,000** (현재 1,975 → +8,025) |
| 🔨 **작업** | CVAT로 크롭+값 대량 확보 / 자동분해 1차 → 사람이 필드 단위 검수·수정 / 부족분 합성(`synth_elements.py`) 보강 (**실데이터 우선**) |
| 📦 **산출물** | `data/elements` ≥1만 쌍 (**클래스별 최소 절대량** 충족) |
| ✅ **완료기준** | 총 ≥1만 + 소수 클래스(GD&T·Roughness ≥1,500 / Datum ≥1,000 / Hole ≥800) 달성 |
| 📄 **상세** | [`데이터라벨링 작업계획`](Element_Donut_데이터라벨링_작업계획.md) |

#### ▷ Phase 2 — 구조화 정합  · 레버 ② · *질 다듬기*

| | |
|---|---|
| 🎯 **목표** | 라벨의 필드 분해를 **정확화**(자동파싱 노이즈 제거) |
| 🔨 **작업** | `pending` 1,974건 손검수와 **동시에** 필드 분해 교정 / 자동파싱 실패(`Ø15H7`, Hole 등) 사람 교정 |
| 📦 **산출물** | 검증된 구조화 라벨(`status=reviewed`) |
| ✅ **완료기준** | 형식검증 통과율 ≥99% + 소수 클래스 2인 검수 완료 |

#### ▷ Phase 3 — 재학습 · 평가  · *성과 확인*

| | |
|---|---|
| 🎯 **목표** | 진전을 **정량 확인** |
| 🔨 **작업** | 재학습 → **field-level P/R/F1**(부분점수)로 추적, **실검증셋으로만** 측정 |
| 📦 **산출물** | 학습된 모델 + F1 리포트 |
| ✅ **기대** | 데이터가 ~1만에 근접할수록 **F1 0.9대** 도달 가능 |

## 9. 한눈에 보는 체크리스트

- [ ] 클래스별 필드 스키마 동결 (Hole_Callout 신설, 누락 필드 보강)
- [ ] CVAT 라벨에 필드별 text attribute 구성
- [ ] `parse_to_schema` 자동분해 1차 적용 → 초안 생성
- [ ] `pending` 라벨 손검수 + 필드 단위 교정 (자동분해 실패 케이스 교정)
- [ ] 실 라벨 **≥ 1만** 확보 (+ 합성 보강)
- [ ] 재학습 → **field-level F1**(부분점수)로 진전 추적, 실검증셋 평가

> **요약**: 값을 의미 필드로 쪼개 **사람이 정확히** 라벨링 = 노이즈 제거 + 부분점수 + 출력 안정화(**②**).
> 단, 이것만으로는 부족하다 — **실 라벨 ≥ 1만 확충(①)** 이 논문 수준에 닿는 **결정적 레버**이며,
> **②로 틀을 잡고 ①로 채우는** 순서로 함께 가야 사용가능 수준에 도달한다.
