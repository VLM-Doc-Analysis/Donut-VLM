# 구조화 스키마 수작업 Annotation 가이드

> **한 줄 정의** — 요소 크롭의 값을 **통짜 텍스트**(`"Ø15H7"`)가 아니라 **의미 단위로 쪼갠 필드**
> (`{nominalValue:"Ø15", toleranceClass:"H7"}`)로, **정규식 자동이 아니라 사람이** 라벨링·검수하는 작업.
>
> 📍 **이 문서의 위치**: 성능 개선 **레버 ②(질)** 를 다룬다.
> 짝이 되는 **레버 ①(데이터 확충, 양)** → [`데이터라벨링 작업계획`](Element_Donut_데이터라벨링_작업계획.md),
> 둘의 관계·순서 → **§8 로드맵**.

**읽는 순서**: 왜 필요한가(§1) → 무엇인가(§2) → 지금 방식과 한계(§3·§4) → 효과(§5) → 작업법(§6) → **기호 표현(§7)** → 큰 그림(§8) → 체크리스트(§9).

> ⚠️ **2026-06-29 정정**: 본 문서 곳곳의 **"실 라벨 ≥1만 확충이 결정적 레버 ①"** 전제는 두 논문 원문
> 검증으로 **오류로 판명**됐다. F1 0.963 을 낸 *Multi-Stage Hybrid*(arXiv:2510.21862)는 **1,406 크롭**
> (우리 1,975보다 **적음**)으로 달성했고, "13,000 / 1/7" 수치는 두 논문 어디에도 없다. 따라서 **양(≥1만)이 아니라
> 라벨/크롭 품질(이 문서가 다루는 "질" = 레버 ②)과 희소 클래스 절대량이 진짜 레버**다. 아래 "≥1만" 목표 수치는
> 상한 야심치로만 읽고, 우선순위는 **품질 정제 → 희소 클래스 보강**이다. (재진단: [`성능미달_원인_및_해결방안`](Element_Donut_성능미달_원인_및_해결방안.md))

# Part I. 구조화 스키마 이해하기

## 1. 왜 필요한가 — "거의 맞은 답"이 0점으로 묻힌다

Donut 값 인식 결과를 **"정답과 완전히 똑같은가(exact)"** 로만 채점하면 **정확도 0%** — 사용 불가처럼 보인다.
하지만 **글자 단위(char 유사도)** 로 보면 **약 0.47** — 모델은 실제로 **절반쯤 읽고 있었다.**

즉 **"거의 맞았지만 한두 글자 틀린" 답까지 전부 0점** 처리되어 **진전이 가려진** 것이다.

| 정답 | 모델이 읽은 값 | exact 채점 | 실제 상태 |
|---|---|---|---|
| `Ø6.5` | `Ø6.4` | ❌ 0점 | 거의 맞음 (한 글자 차이) |
| `90±0.1` | `90` | ❌ 0점 | 핵심값은 맞고 공차만 놓침 |
| `25` | `5` | ❌ 0점 | 앞자리 누락 |

> **문제의 핵심**: "전부 맞아야 1점"인 채점은 **부분 정답을 못 본다.**
> 값을 의미 필드로 쪼개면 이 **부분 진전을 점수로 살릴 수 있다** — 그래서 구조화 스키마가 필요하다.
> *(실측: exact `0.000` vs char 유사도 `0.468` — 같은 모델인데 채점 방식으로 평가가 갈렸다.)*

---

## 2. 무엇인가 — 값을 "필드 dict"로 쪼갠 것

값 문자열을 **카테고리별 필드 dict**로 분해한 형태다. (논문 Numerical VLM 방식)

```
"8X Ø6.5±0.1"  →  { "quantity":     "8",       ← 수량
                    "nominalValue": "Ø6.5",    ← 기준값
                    "upperLimit":   "+0.1",    ← 위 공차
                    "lowerLimit":   "-0.1" }   ← 아래 공차
```

클래스별 필드 정의 (현재 `parse_to_schema` 기준):

| 클래스 | 필드 |
|---|---|
| Dimension | `quantity`, `nominalValue`, `upperLimit`, `lowerLimit` |
| GD&T_FCF | `geometricCharacteristic`, `tolerance`, `datumReference[]` |
| Surface_Roughness | `roughness` |
| Datum | `datum` |
| Hole_Callout | ⚠️ **스키마 없음 → 통짜 `value`** (신설 필요, §6 참고) |

---

## 3. 지금은 어떻게 하나 — 자동 분해 vs 수작업

지금도 구조화는 **하고 있다. 다만 정규식이 추측한다.** 제안은 그 추측을 **사람이 책임지게** 하는 것이다.

| 구분 | 현재: **자동 분해** | 제안: **수작업 구조화** |
|---|---|---|
| 라벨 원천 | CVAT `value` 텍스트 **1칸** (`"8X Ø6.5±0.1"`) | CVAT에서 **필드별로 입력** (`quantity=8, nominal=Ø6.5, ±=0.1`) |
| 분해 주체 | 학습 시 `parse_to_schema` **정규식**이 추측 | **사람**이 분해(또는 자동분해를 검수·수정) |
| 정확성 | 규칙 못 맞으면 **조용히 오라벨** | 사람이 보장 |
| 결과 | "대충 맞는" 노이즈 정답 | 검증된 정답 |

---

## 4. 자동 분해는 어디서 새는가 — 실제 실패 사례

실제 라벨에 `parse_to_schema`를 돌려보면 대부분 맞지만, 다음이 샌다:

| 원본 값 | 자동분해 결과 | 무엇이 문제인가 |
|---|---|---|
| `Ø15H7` | `{nominalValue:"Ø15H7"}` | **H7(끼워맞춤 공차등급) 분리 안 됨** — 정답은 `nominalValue:Ø15` + `toleranceClass:H7` |
| `M6X1.0 ⏥ .50` (Hole) | `{value:"M6X1.0 ⏥ .50"}` | **스키마 없어 통째로** — `thread:M6X1.0` + `counterbore:.50` 분리 안 됨 |
| `- 'A'±0.001 -` | `{upper:+0.001, lower:-0.001, nominalValue:"- 'A' -"}` | **쓰레기 입력**인데 규칙이 억지로 구조 생성 → 사람이면 걸러냄 |
| `.03 x 45°` | `{nominalValue:".03 x 45°"}` | `x`가 "수량 곱"인지 "모따기 by"인지 **모호** (`8X…`는 수량 처리 — 케이스마다 규칙이 다르게 동작) |

> 이 노이즈가 **그대로 학습 정답**으로 들어가 모델 품질을 깎는다. → 사람이 메워야 하는 지점.

---

## 5. 무엇이 좋아지나 — 4가지 효과

| 효과 | 설명 |
|---|---|
| **① 부분 점수(partial credit)** | field-level P/R/F1이라 `nominalValue`만 맞고 `tolerance` 틀려도 **0.5점**. exact 0.000에 묻히던 "절반의 진전"이 점수로 드러나 **학습 신호·평가가 매끄러워짐** |
| **② 출력 안정화** | 정답이 항상 같은 필드 구조 → 모델이 배우기 쉽고 hallucination↓ *(논문: "schema-constrained learning이 출력 변동성을 줄였다")* |
| **③ 라벨 정확성** | `Ø15H7`·Hole·쓰레기값 같은 자동분해 실패를 사람이 바로잡음. **라벨 품질 = 모델 품질** |
| **④ 다운스트림 즉시 활용** | `nominalValue=15, tolerance=H7`로 나오면 CAD/검사 DB가 바로 검증 가능. 통짜 `"Ø15H7"`은 또 파싱해야 함 |

## 6. 어떻게 작업하나 — 비용·워크플로

- **라벨링 시간↑**: 크롭당 텍스트 1칸 → 필드 여러 칸.
- **CVAT 지원**: 박스 하나에 **여러 text attribute** 부여 가능(예: `quantity`, `nominal`, `upper`, `lower`).
- **스키마 설계 선행**: 클래스별 필드 정의 확정. 특히 통짜인 **Hole_Callout 스키마 신설** 필요 — 예:
  ```
  "4X M6X1.0 ⏥ .50"  →  { "quantity":"4", "thread":"M6X1.0", "counterbore":".50" }
  ```
  (Dimension에는 `toleranceClass`(H7 등) 필드도 추가)
- **현실적 절충 (가장 빠른 길)**: **자동분해를 1차로 돌린 뒤 사람이 검수·수정.**
  처음부터 손으로 다 쪼개는 것보다 빠르고, `pending` 1,975건 검수 단계에 그대로 통합된다.

## 7. 토크나이저 관점 — 공학 기호를 어떻게 표현할까 (핵심 주의점)

값을 필드로 쪼개도(§2), **필드 값 안의 공학 기호**(`Ø ⊥ ⌖ ± ° Ⓜ …`)를 **학습 타깃 시퀀스에 어떻게 적느냐**가 따로 중요하다.

> 📌 **먼저 알아둘 것 (우선순위)**: 기호를 어떻게 표현하느냐는 **성능의 1차 병목이 아니다.**
> 기호 인식 문제는 이미 거의 해소된 상태라, **지금 가장 급한 건 학습 데이터를 늘리는 일**이다.
> 아래 (B)로 바꾸는 것은 **데이터를 키우는 작업과 함께 진행할 품질 개선**으로 보면 된다 — 이것만 바꿔서는 성능이 크게 오르지 않는다.

> **왜 문제가 되나 (쉽게)**: AI는 글을 읽을 때 **'토큰'이라는 작은 글자 조각**으로 나눠 본다.
> 이 조각 사전은 **흔한 글자·단어** 위주라, 도면 기호처럼 **드문 기호는 사전에 없다.**
> 사전에 없는 기호는 AI가 못 알아보고 **잘게 부서지거나 "모르는 글자"로 버려진다** → 그 값을 못 배우거나 깨진 결과가 나온다.

표현 방식은 크게 **세 가지**다.

### (A) 기호를 사전에 새 단어로 등록

도면 기호(`⌖ ⊥ Ⓜ …`)를 AI의 글자 사전에 **그대로 새 항목으로 추가**한다.
- 👍 출력이 **사람이 보는 도면 표기와 똑같다.**
- 👎 비슷하게 생긴 기호끼리 **헷갈리기 쉽고**, 시스템이 글자를 자동 정리하는 과정에서 **변형될 수 있다** — 특히 동그라미 안 글자(`Ⓜ Ⓛ`)는 평범한 `M L`로 풀어져 섞인다.

### (현재 프로젝트) 기호를 '코드 문자열'로 바꿔 적기

기호를 `Ø → "U+00D8"`처럼 **영문·숫자 코드로 변환**해 적고, 마지막에 다시 기호로 되돌린다.
- 👍 사전에 없는 기호 문제·자동 변형을 **안전하게 피한다.**
- 👎 글이 **길어지고**, AI가 `U+00D8` 같은 **코드를 통째로 정확히 외워야** 한다 — 실제로 실험에서 `U21`처럼 **반쯤 깨져** 나오기도 했다.

### (B) 기호를 '뜻 단어'로 바꿔 적기 ★**권장**

기호를 **그 의미를 나타내는 영어 단어**로 적는다. 예: `⌖ → position`, `Ⓜ → MMC`.

```
<fcf>
  <characteristic>position</characteristic>   ← ⌖ 의 뜻
  <tolerance>0.05</tolerance>
  <modifier>MMC</modifier>                     ← Ⓜ 의 뜻
  <datum>A</datum><datum>B</datum><datum>C</datum>
</fcf>
```
- `position`, `MMC` 같은 단어는 **AI가 이미 아는 흔한 단어** → 새로 외울 부담이 적다.
- 기호가 **폰트마다 다르게 그려져도** 단어는 그대로라 **흔들리지 않는다.**
- 마지막에 **단어 ↔ 기호 변환표**만 두면 다시 기호로 되돌릴 수 있다.

### 한눈에 비교

| 방식 | 모르는 기호 문제 | AI가 배우기 | 폰트 달라도 강한가 |
|---|---|---|---|
| (A) 기호 그대로 등록 | ⚠️ 변형 위험 | 보통 | 약함 |
| 코드 문자열 (현재) | ✅ 안전 | 보통 (코드 암기) | 보통 |
| **(B) 뜻 단어 ★** | ✅ 안전 | **쉬움 (아는 단어)** | **강함** |

> **권장**: **(B) 뜻 단어로 학습**시키고, **변환표로 마지막에 기호 복원**하는 방식이 가장 깔끔하다.
> - 예: `Ⓜ ↔ MMC`, `⌖ ↔ position`, `⊥ ↔ perpendicularity`, `Ø ↔ diameter`.
> - (B)는 **구조화 스키마(§2)와 자연스럽게 합쳐진다** — "값을 필드로 쪼개고, 기호도 뜻으로 적는" 같은 일이기 때문.
>
> 기호를 사전에 추가하는 구체 절차는 [`토크나이저 기호추가 가이드`](Element_Donut_토크나이저_기호추가_가이드.md) 참고.

# Part II. 큰 그림에서의 위치

## 8. 데이터 확충과 함께 — 두 레버와 로드맵

### 먼저, 이건 만능이 아니라 "보완재"다 ⚠️

구조화 수작업 annotation(②)만으로는 부족하다.

- 0% 벽의 **1차 원인**은 데이터 **양**이 아니라 **품질**(노이즈 라벨·비정렬 복합 크롭) — 논문 B 는 1,406 크롭으로 0.963.
- ②는 적은 데이터에서 **신호를 더 짜내고 평가를 등급화**하지만,
  사용가능(exact > 0)에 닿으려면 **데이터 증량(①)** 과 **함께** 가야 한다. → 그래서 레버가 **둘**이다.

### 두 핵심 레버 (① 양 × ② 질)

| # | 레버 | 무엇을 하나 | 왜 중요한가 | 비용 |
|---|---|---|---|---|
| **①** | **라벨/크롭 품질 정제 + 희소 클래스 보강** | 깨진/오라벨 손수정·단일기호 정렬 크롭 + GD&T·Hole 등 절대량 확보 | 논문 B 는 **1,406 크롭(우리보다 적음)** 으로 F1 0.963 → 양보다 **품질**이 레버 | 중~높음 |
| **②** | **수작업 구조화 스키마** | 값을 **의미 필드로 분해** | **부분점수 + 출력 안정** + **자동파싱 노이즈 제거** | 중~높음 |

- **①만** 하면: 데이터는 많아지지만 라벨이 통짜/노이즈면 학습이 흔들린다.
- **②만** 하면: 라벨은 깨끗해도 표본이 적어 OCR을 못 배운다 — 실험에서 **3클래스로 줄여도 0% → 0%** 였다.
- **①(양) × ②(질)** 가 곱해져야 논문 수준에 닿는다.

### 왜 이 순서인가 — "②로 틀을 잡고 ①로 채운다"

스키마(②의 필드 정의)를 **먼저 동결**한 뒤 대량 라벨링(①)을 해야 한다.
스키마 없이 1만 장을 통짜로 모으면 나중에 **전부 재분해**해야 하기 때문이다.

### 단계별 로드맵 (Phase 0 → 3)

각 단계는 **앞 단계 산출물이 있어야 시작**된다(완료기준 = 다음 단계 진입 조건).

```
 Phase 0          Phase 1 ★결정적       Phase 2          Phase 3
 스키마 확정  ──▶  데이터 확충 ≥1만  ──▶  구조화 정합  ──▶  재학습 · 평가
 (레버 ②)         (레버 ①)            (레버 ②)         (확인)
 "틀 만들기"       "양 채우기"          "질 다듬기"       "성과 확인"
```

> 흐름의 논리: **②로 틀 → ①로 양 → ②로 질 → 평가.**
> ①(데이터 확충)이 **결정적 레버**, ②(구조화 스키마)가 그 **앞뒤에서 틀과 질을 받친다.**

#### ▷ Phase 0 — 스키마 확정 · 레버 ② · *틀 만들기*

| | |
|---|---|
| 🎯 **목표** | 라벨 구조를 **먼저 고정**해 이후 재작업을 없앤다 |

## 9. 한눈에 보는 체크리스트

- [ ] **(Phase 0)** 클래스별 필드 스키마 동결 — Hole_Callout 신설, `toleranceClass` 등 보강
- [ ] **(Phase 0)** CVAT 라벨에 필드별 text attribute 구성
- [ ] **(Phase 1)** `parse_to_schema` 자동분해 1차 적용 → 초안 생성
- [ ] **(Phase 1·2)** `pending` 라벨 손검수 + 필드 단위 교정(자동분해 실패 케이스)
- [ ] **(Phase 1)** 실 라벨 **≥ 1만** 확보 (+ 합성 보강)
- [ ] **(Phase 3)** 재학습 → **field-level F1**(부분점수)로 진전 추적, 실검증셋 평가

> **요약**: 값을 의미 필드로 쪼개 **사람이 정확히** 라벨링 = 노이즈 제거 + 부분점수 + 출력 안정화(**②**).
> 단, 이것만으로는 부족하다 — **실 라벨 ≥ 1만 확충(①)** 이 논문 수준에 닿는 **결정적 레버**이며,
> **②로 틀을 잡고 ①로 채우는** 순서로 **함께** 가야 사용가능 수준에 도달한다.
