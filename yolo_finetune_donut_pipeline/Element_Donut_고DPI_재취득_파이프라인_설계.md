# Element Donut — 고DPI 재취득 파이프라인 설계 (확정)

> 목적: element 크롭의 **해상도·블러 천장**을 푸는 고품질 재취득 파이프라인.

## 0. 목표 & 제약

- **목표**: 134px·블러(69%) 크롭 → **고해상·정렬 단일요소 크롭**으로 재생성(품질 천장 제거).
- **제약(확인됨)**:
  - 기존 1,975 값크롭 = 외부 "Download" → **재취득 불가**(소스·박스좌표 없음).
  - 로컬 소스 = **PDF 50개**뿐. GT 검출 박스 = view(AABB, 정규화) + element(OBB, 정규화) — **DPI 무관**.
  - 학습된 YOLO 가중치 없음(runs/ gitignored).
- ⚠️ **Phase 1 은 50개 도면에 한정** → 양·GD&T(박스 5개) 부족은 Phase 2(소스 확충)에서 해결.

## 1. 전체 흐름

```
raw_pdf/*.pdf (50)
 └(1) 래스터화 @600 DPI            page.png (2480×3509 → 4960×7018)
   └(2) view 박스 적용             page 좌표 정규화 AABB (GT 재사용 / Phase2: YOLO-det)
     └(3) view 크롭 @600 DPI
       └(4) element OBB 적용        view 좌표 정규화 OBB (GT 재사용 / Phase2: YOLO-obb)
         └(5) rectify_obb           정렬 단일요소 크롭 — ★native 고해상 유지(고정높이 금지)
           └(6) 값 라벨링           {raw, fields(Unicode)} — Donut pre-fill + 사람 교정
             └ 고품질 학습쌍 (data/elements_hidpi/{images,labels})
```
- 좌표가 모두 **정규화**라 (1)의 DPI만 바꾸면 (2)~(5)는 그대로 고해상으로 따라옴.

## 2. 확정 결정

| 항목 | 결정 | 근거 |
|:--|:--|:--|
| **래스터 DPI** | **PDF별 적응형**: 벡터 = **600**(or 900), 스캔 = **min(600, native)** | 소스 검토(§7) 결과 50개 중 **벡터 52% / 스캔 48%(native 중앙 300)**. 스캔을 균일 600 으로 올리면 업스케일=블러 증폭이라 native 상한 적용 |
| **크롭 사이징** | rectify **native 해상도 유지**, 고정높이 금지 | 외부 도구의 134px 정규화가 천장의 원인 |
| **Donut 입력** | `image_size` 448 → **768** | 고해상 크롭의 디테일 보존(현재 448은 우리가 축소해 둔 값) |
| **라벨 스키마** | `{raw, fields(Unicode 표준)}` | [`라벨_타깃_분리_설계`](Element_Donut_라벨_타깃_분리_설계.md) |
| **학습 타깃** | `TARGET_MODE` 토글 — 지금 **flat**, ~1만+ 시 structured | 소량엔 flat 우위(0.444>0.349) |

## 3. 2-Phase 전략

### Phase 1 — 품질 파일럿 (50 PDF, 박스 재사용)
- 600 DPI 재래스터 + **기존 GT 박스(view/element) 적용** → rectify → **~1,278 고DPI 크롭**. **YOLO 추론 불필요.**
- 값 라벨링: 현 Donut 으로 `raw` pre-fill → 사람 교정(+ `fields` 분해).
- 🎯 **부가가치 = 통제된 품질 A/B**: 같은 크롭을 300 vs 600 으로 만들어 학습/평가 → **품질(DPI) 단독 효과를 분리 측정**(양 고정). **★ 벡터 PDF 26개에 한정**(스캔은 native 상한이라 600 효과 없음 — §7).
- ⚠️ 한계: 양은 현재와 비슷(~1,278), GD&T 5개 → **양·GD&T 미해결**. "품질 레버 검증"이 목적.

### Phase 2 — 양·균형 확장 (소스 확충, 사용자 의존)
- **GD&T·Hole 풍부한 도면 PDF 추가 소싱**(~수백 장) → 600 DPI 래스터.
- **YOLO view/det + element/obb 재학습**(데이터셋 존재) → 신규 페이지 검출 → rectify.
- CVAT 로 박스+값(`fields`) 라벨링. 목표 ~1만+ (희소 클래스 집중).
- 이때 `TARGET_MODE=structured` 전환(데이터 충분).

## 5. 즉시 실행 가능한 첫 스텝 (Phase 1)

1. `rasterize_pdf.ipynb` 의 `DPI=300` → **600** 으로 50개 재래스터.
2. GT 박스(view→element, 정규화)로 600 DPI 크롭 추출 스크립트(`crop_utils.rectify_obb` 재사용) → `data/elements_hidpi/`.
3. 300 DPI 동일 크롭도 생성(A/B 짝).
4. 값 라벨 이식(기존 stem 매칭 가능분) + 부족분 라벨링.
5. flat 학습 → **600 vs 300 field-F1 비교** → 품질 레버 정량 확정.

## 6. 기대효과 / 검증

- **검증**: Phase 1 의 600 vs 300 A/B 가 양을 고정한 채 **품질 효과를 단독 측정**(지금껏 없던 통제 실험).
- **기대**: 600 DPI 가 charsim/exact 를 끌어올리면 → 품질 천장이 실재함을 입증, Phase 2(양+품질) 투자 정당화.
- **주의**: 효과가 미미하면 병목이 해상도보다 **블러 자체(소스 스캔 품질)** 일 수 있음 → 디블러/소스 재취득으로 방향 전환.

## 7. 소스 PDF 검토 결과 (2026-06-29, `data/raw_pdf` 50개)

`fitz` 로 벡터 path 수 + 임베드 이미지 유효 DPI 측정 (`scratchpad/pdfscan.py`):

| 종류 | 개수 | 600 DPI 효과 |
|:--|--:|:--|
| **벡터(CAD, 해상도 무관)** | **26 (52%)** | ✅ 확실(600/900 선명) |
| 스캔 350~550 DPI | 8 (16%) | △ native 까지만 |
| 스캔 250~350 DPI | 10 (20%) | ✗ 거의 없음 |
| 스캔 <250 DPI | 5 (10%) | ✗ 없음(업스케일) |
| 스캔 ≥550 | 1 (2%) | 이미 고해상 |

- 래스터(스캔) 24개의 **유효 DPI 중앙값 = 300**.
- **결론**: ① 벡터 52% 는 600 DPI 로 진짜 선명 → 파이프라인 가치 확실(현 블러 일부는 *벡터를 저DPI 래스터*한 탓). ② 스캔 48% 는 native ~300 이라 600 무의미(특히 <350=30% 는 블러 증폭) → **균일 600 금지, 적응형 DPI**. ③ Phase 1 A/B 는 **벡터 26개에 집중**. ④ 스캔 ~30% 의 품질 천장은 **원본 재스캔(사용자 측)** 외엔 불가.
- **첫 확인**: 벡터 PDF 1개를 600 으로 렌더해 크롭 육안 확인 → 선명도 향상 즉시 검증 가능.
