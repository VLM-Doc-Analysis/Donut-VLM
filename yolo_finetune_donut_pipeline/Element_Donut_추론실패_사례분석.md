# Element Donut — 추론 실패 사례 분석

> 대상: `checkpoints_elements/final` (토크나이저 확장 후 재학습본, 2026-06-24)
> 추론: `donut_training_elements.ipynb` Step 6 `predict()` · 환경 `kardi_env`
> 작성: 2026-06-24
> 관련: [`Element_Donut_평가리포트.md`](Element_Donut_평가리포트.md) · [`Element_Donut_토크나이저_기호추가_가이드.md`](Element_Donut_토크나이저_기호추가_가이드.md)

검증셋 추론에서 나온 두 가지 비정상 출력의 원인을 raw 토큰열 수준에서 분해한다.
**공통 결론: 토크나이저 확장(기호·숫자 OOV 해결)은 정상 작동했고, 남은 실패는 "약한 모델 + 어려운 크롭"** 이다.

---

## 사례 ① 퇴화 반복 + 비종료 (Dimension)

- 입력: `val/images/1375_t45_f001_s01_Dime.png` (복합 공차 `−⌀.750 (+.000 / −.001)`)
- GT: `{"Dimension": "-Ø.750+000-001"}`

```
[greedy(이전)]  토큰 129개, max까지 감
RAW: <s_element><s_Dimension>Ø.5 +0000000…(0이 79개)… - 1 - 1 - 1 …</s>

[반복억제 적용]  토큰 12개, 정상 종료
RAW: <s_element><s_Dimension>Ø.5 +0 - 01</s>
```

### 원인

1. **직접 트리거 = greedy 디코딩의 자기강화 반복.**
   약한 디코더가 `0` 토큰에 한 번 높은 확률을 주면, 자기회귀 특성상 같은 토큰을 계속 선택 →
   `00000…`, `- 1 - 1 …` 루프. 종료(EOS) 신호가 약해 128 토큰 한계까지 감.
   → `no_repeat_ngram_size=3, repetition_penalty=1.5` 만 켜도 12토큰에서 깔끔히 종료(증명됨).
2. **근저 = 인식 부정확.** 반복을 없애도 `Ø.5 +0 - 01` 로 GT와 다름.
   공차가 위아래로 쌓인 작은 글씨를 `.750→.5`, `+.000→+0`, `-.001→-01` 로 축약·오독.
   자리수 반복 패턴(`000`,`001`)에서 길이 카운팅을 못하는 건 약한 디코더의 전형.
3. **토크나이저 확장의 부수효과(정상).** `0`·`1` 이 독립 토큰이 돼(이전엔 묶임) 같은 반복이라도
   토큰 단위로 더 길게·도드라지게 보임. 수정 자체는 옳고, 반복의 원인이 아니라 가시성만 키움.

### 조치

- **즉효(반영 완료):** `predict()` 의 `generate()` 에 `no_repeat_ngram_size=3, repetition_penalty=1.5` 추가
  → 끝없는 반복·비종료 제거. **단 증상 치료일 뿐 값 정확도는 안 올라감.**
- **근본:** 데이터 확충·재학습 (복합 공차는 데이터 부족/불균형이 원인).

## 사례 ② 값 오독: 글자 `A` → 따옴표 `"` (Datum)

- 입력: `val/images/1412_t45_f005_s00_Datu.png` (문자 `A` + 데이텀 삼각형 지시자)
- GT: `{"Datum": "A"}` · 출력: `{"Datum": "\""}`

```
RAW    : <s_element><s_Datum> "</s_Datum></s_Dimension></s>
토큰    : [<s_element>] [<s_Datum>] [▁"] [</s_Datum>] [</s_Dimension>] [</s>]
첫 스텝 : <s_Datum>=0.715  <s_GD&T_FCF>=0.184  <s_Dimension>=0.048 …
```

### 원인 — "타입은 맞고, 값 읽기에서 실패"

1. **타입 분류는 정상.** 첫 스텝에서 `<s_Datum>` 0.715 선택 → 이전 평가의 "전부 Dimension 붕괴"보다 개선.
   문제는 그다음 **값(value) 자리**.
2. **값 자리에서 `A` 대신 `"`(따옴표) 출력 — 핵심 실패.**
   - 이 크롭은 깨끗한 글자 1개가 아니라 **데이텀 기호 그래픽 전체**(문자 `A` + 빨간 삼각형 지시자
     + 리더선 + `A` 왼쪽의 작은 틱 마크)를 통째로 담고 있다.
   - 인식 대상 "A"가 그래픽에 묻혀 있고, 모델은 **`A` 왼쪽 틱/리더 마크를 따옴표로 오인**
     (틱·아포스트로피·따옴표는 시각적으로 유사).
   - 즉 OBB/rectify 크롭이 **문자가 아닌 기호 도형 중심**이라 난이도가 높다(Datum류 공통 약점).
3. **구조(태그)도 미완성.** `<s_Datum>` 로 열고 엉뚱하게 `</s_Dimension>` 로 닫음(서로 다른 타입).
   `token2json` 이 `<s_Datum>…</s_Datum>` 를 먼저 매칭해 파싱은 됐지만, **태그 짝맞춤을 깔끔히
   학습하지 못한 약한 모델**이라는 증거.
4. **근저.** 소수 클래스(train Datum 138장, 그중 "A" 52장)에서 과적합된 약한 모델.
   최빈값 "A" 인데도 그래픽 노이즈에 흔들림.

### 조치 (반복억제로는 해결 안 됨 — 인식 오류)

| 층위 | 조치 |
|---|---|
| **데이터/크롭** | Datum 크롭이 삼각형 지시자·리더까지 포함 → 검출 박스를 **문자 영역 중심으로 좁히거나** 데이텀 문자만 떼는 전처리. 인식 난이도를 직접 낮추는 가장 효과적 방법. |
| **학습/데이터량** | Datum 등 소수 클래스 샘플 확충 + 클래스 균형. 그래픽 노이즈가 많은 타입일수록 더 많은 예시 필요. |
| **구조 안정화** | 재학습으로 태그 짝맞춤(`</s_Datum>`) 일관성 확보. |

## 종합

| | 사례 ① (반복) | 사례 ② (오독) |
|---|---|---|
| 증상 | 끝없는 `0…`/`- 1` 반복, 비종료 | 글자 `A` → `"` |
| 타입 토큰 | 맞음(Dimension) | 맞음(Datum, 0.715) |
| 직접 원인 | greedy 자기강화 반복 | 그래픽 위주 크롭 오독 |
| 디코딩으로 해결? | **예**(반복억제) | 아니오 |
| 근본 해결 | 데이터·재학습 | 크롭 품질·데이터·재학습 |

- **토크나이저 확장은 검증됨** — 두 사례 모두 `Ø`, 숫자, 기호가 `<unk>` 없이 출력됨.
- 남은 병목은 **모델 품질(소규모·불균형 데이터, 과적합)** 과 **크롭 품질(기호 그래픽이 글자를 덮음)**.
- 반복억제는 사례 ① 같은 디코딩 증상에만 유효하고, 값 정확도(사례 ②)는 데이터·학습으로만 개선된다.
