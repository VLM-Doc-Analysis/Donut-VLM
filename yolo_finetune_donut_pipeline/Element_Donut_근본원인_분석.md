## 2. 근본 원인

### ① 타입 분류 ≠ 값 읽기 — 난이도가 본질적으로 다름
- **타입** = 이미지 전역 특징으로 **1회 5지선다** → 1800장으로도 학습됨(91%).
- **값** = 디코더가 이미지를 **훑으며 글자를 순서대로 디코딩**(OCR) → 정밀 cross-attention 필요 →
  미학습. 그래서 **"첫 글자 + 닫기"** 라는 퇴화 정책으로 수렴.

### ② 데이터 절대 부족 + 불균형 + 품질
> ✅ **2026-06-29 (논문 §3 확정)**: 논문 numeric VLM 은 **~11,000–13,000 크롭**으로 학습(Multi-Stage ~13,000→0.963,
> From Drawings 11,469→0.935). **우리는 1,975(≈1/7).** (초판의 "1,406 크롭" 은 *도면* 수 오독이라 정정 — 양은 명백한 격차.)
- **데이터 규모**: ~2k 로는 도메인 "읽기"를 from-scratch 로 못 배움(값-전용 train 정확도 12% = underfit).
- **클래스 불균형**: 65% Dimension, GD&T 180(논문 3,215)·Hole 70 — 희소 클래스 학습 방해.
- **품질**: 블러 69%·134px·노이즈([`이미지품질_분석`](Element_Donut_이미지품질_분석.md)) — 양과 함께 끌어올려야 함.

### ③ 도메인 격차
- `donut-base` 는 문서/영수증 텍스트로 사전학습. 도면 기호·세로로 쌓인 공차·작고 노이즈 많은
  크롭과는 거리가 멀어 읽기 prior 가 잘 전이되지 않음.

### ④ teacher-forcing loss(0.20)가 실패를 숨김
- 손실의 대부분은 **쉬운 구조 토큰**(`<s_Dimension>`·닫힘·eos)에서 나오고, 정작 **값 토큰은 못 맞춤**.
- 그래서 train_loss 는 낮은데 free-generation 정확도는 train 조차 ~2%. (loss 와 실제 성능의 괴리)

### ⑤ (부수) per-char 토큰화로 값 시퀀스가 길어짐
- 숫자/기호를 글자당 1토큰으로 추가(기호 OOV 해결엔 **필수**)하면서, 약한 디코더가 여러 토큰을
  순서대로 맞추기 더 어려워짐. (원인이라기보단 난이도 가중 요인)

### 학습 곡선 (과적합 정황)
```
eval_loss: 4.48 → 1.06(ep2.2) → 0.67(ep7, best) → 0.67~0.78 정체  [조기종료 ep9.4]
train_loss: → 0.20    # train-eval 괴리 + 값은 train도 못 읽음
```

## 3. 해결책 (영향 큰 순)

| # | 해결책 | 왜 효과적 | 비용 |
|---|---|---|---|
| **1** | **합성 데이터 대량 생성** | 폰트로 치수·공차·데이텀 텍스트 수천~수만장 렌더링(노이즈/회전/배경 augment) → **순차 읽기를 직접 학습**. 라벨 공짜. OCR 미세조정의 정석. | 중 |
| **2** | **OCR 사전학습 인식기로 교체/초기화** | 값 읽기를 donut-base 대신 **TrOCR**(대규모 OCR 사전학습) 등으로. YOLO가 이미 타입을 주므로 값 텍스트만 인식 → 소량 데이터에도 전이 잘 됨. | 중 |
| **3** | **타입 생성 제거(값 전용 학습)** | YOLO가 타입을 알므로 Donut은 **값만** 출력 → 디코더 용량을 읽기에 집중. 알려진 타입은 디코더 프롬프트로 강제. | 소 |
| **4** | **크롭 품질 개선** | 검출 박스를 **텍스트 영역으로 좁혀** 데이텀 삼각형·리더선 등 그래픽 노이즈 제거 → 값 자리 오독(예 `A→"`) 직접 완화. | 중 |
| **5** | **불균형/길이 보정 + 강한 augment** | 다중글자·소수클래스 oversample, 대비/스케일 augment → 퇴화 정책 붕괴 완화·과적합 억제. | 소 |
| **6** | **해상도/종횡비** | 넓고 낮은 치수 크롭을 384 정사각으로 늘리면 왜곡 → **종횡비 보존 패딩** 또는 더 넓은 입력. | 소 |

### 권장 경로
1. **먼저 ① 합성 데이터 + ③ 값 전용 학습** — 이 둘만으로 "읽기" 학습이 가능해질 가능성이 높다(최단·최저비용).
2. **그래도 부족하면 ② OCR 사전학습(TrOCR) 전환** — 본질적 해법. 소량 실데이터로도 읽기가 전이된다.
3. ④⑤⑥ 은 위와 **병행하는 보조 개선**.

## 4. 무엇이 문제가 "아닌가" (배제된 가설)

- **반복억제 파라미터 탓이 아님** — 억제 없는 greedy 에서도 동일하게 첫 글자만 읽음.
- **토크나이저 미수용(OOV) 탓이 아님** — 확장 후 `Ø`/숫자/기호가 `<unk>` 없이 출력됨(검증 완료).
- **단순 과적합이 아님** — train 값 정확도도 1.7%. 일반화 이전에 **학습 자체가 안 됨**.
- **디코더 조기 종료(정보는 있음)도 아님** — 종료 강제해도 둘째 글자 복원 실패 → 읽기 능력 미학습.

## 5. 재현 (진단 스니펫)

```python
# 타입/값 분해 + train vs val 비교가 핵심.
# 1) 검증셋 추론 후 raw 토큰열에서 first <s_X>(타입)와 <s_X>..</s_X>(값) 분리
# 2) type_ok = (pred_type==gt_type),  val_ok = (norm(pred_val)==norm(gt_val))
# 3) train 서브셋에도 동일 측정 → train도 낮으면 '학습 실패'(과적합 아님)
# 4) min_new_tokens / num_beams 로 종료 강제 → 둘째 글자 복원되는지 확인
```
> 측정 결과: type 91.4% vs value 1.0%, train 1.7% ≈ val 0.8%, 종료 강제해도 둘째 글자 미복원.
