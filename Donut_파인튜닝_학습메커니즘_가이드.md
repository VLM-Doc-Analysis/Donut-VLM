# Donut 파인튜닝 학습 메커니즘 가이드 (원리편)

> **무엇을 다루나**: Donut이 "**이미지 → 구조화 JSON(토큰열)**" 매핑을 **어떻게 학습하는가**를 직관적으로 설명한다.
> 절차(데이터·토큰등록·CFG·실행)는 [`Donut_CORD_v2_파인튜닝_가이드.md`](Donut_CORD_v2_파인튜닝_가이드.md) 참고 — 이 문서는 그 **원리편**.

CORD-v2(영수증)를 예로 들지만, 도면/요소 태스크에도 그대로 적용된다.

---

## 0. 한 줄 직관

학습 = **받아쓰기 시험을 수천 번 반복하며 채점·교정하는 것.**

| 비유 | 실제 |
|---|---|
| 문제지 | 영수증 이미지 |
| 정답지 | 그 영수증의 JSON을 토큰으로 푼 것 (`<s_cord-v2><s_menu>…</s>`) |
| 채점 | 각 칸에서 모델의 "다음 토큰" 예측이 정답과 얼마나 다른가 (Loss) |
| 교정 | 틀린 만큼 가중치를 **아주 조금씩** 수정 (gradient descent) |

---

## 1. 학습 데이터 = (이미지, 정답 토큰열) 쌍

정답 JSON을 **토큰 시퀀스로 변환**해 둔다. 이게 "정답지"다.

```
gt_parse: {"menu":{"nm":"치킨","price":"12000"}, "total":{"total_price":"12000"}}
   │ json2token  (키를 역정렬: top은 total>menu, menu 안은 price>nm)
   ▼
target = <s_cord-v2> <s_total><s_total_price>12000</s_total_price></s_total><s_menu><s_price>12000</s_price><s_nm>치킨</s_nm></s_menu> </s>
         └ task ┘    └──────────────────── 정답 본문 (키는 역정렬) ────────────────────┘                       └ eos ┘
```

> ※ 토큰 **순서**는 `json2token` 이 키를 **역정렬(`sorted(reverse=True)`)** 해 결정한다 — 그래서 `total` 이 `menu` 보다, `price` 가 `nm` 보다 앞에 온다(결정적 순서 보장).

모델의 목표: 이미지를 보고 이 토큰열을 만들어내는 것.

---

## 2. Teacher Forcing — "정답을 보여주며 다음 한 칸 맞히기" ★핵심

학습 때는 모델이 스스로 만든 토큰이 아니라 **정답 토큰을 입력으로** 준다. 각 위치 `t`에서:

> **입력**: 이미지 + 정답토큰[0 … t−1] → **모델 예측**: 위치 t의 "다음 토큰 확률분포" → **정답**: 정답토큰[t]

한 영수증의 여러 위치를 **동시에** 채점한다:

| 지금까지 준 정답 토큰 | 맞혀야 할 다음 토큰 |
|---|---|
| `<s_cord-v2>` | `<s_total>` |
| `…<s_total>` | `<s_total_price>` |
| `…</s_total>` | `<s_menu>` |
| `…<s_menu>` | `<s_price>` |
| `…<s_price>12000</s_price>` | `<s_nm>` |
| `…<s_nm>` | `치` |
| `…<s_nm>치` | `킨` |
| `…치킨` | `</s_nm>` |
| … | … `</s>`(끝) |

> (순서는 §1 처럼 키 역정렬을 따른다 — `total` → `menu`, `menu` 안에서 `price` → `nm`.)

> 직관: 받아쓰기인데 틀려도 **정답 줄을 보여주며** 다음 글자를 묻는다. 그래서 한 글자 틀려도 뒤가 무너지지 않고, **모든 위치를 한 번에 병렬 채점**할 수 있어 학습이 빠르고 안정적이다.

---

## 3. 채점(Loss) + 교정(역전파)

### 채점 — CrossEntropyLoss
각 위치에서 예측 확률분포와 정답 토큰을 비교. 정답에 높은 확률을 줬으면 손실↓, 엉뚱한 데 줬으면 손실↑.
- **패딩 위치는 `-100`** 으로 채점 제외 (빈칸은 안 봄).

### 교정 — 역전파가 두 곳 모두로 흐른다
```
손실 ──역전파──▶ 디코더(mBART) 가중치   : "이전 토큰+이미지 보고 다음 토큰 잘 고르게"
            └──(cross-attention 통해)──▶ 인코더(Swin) 가중치 : "값 생성에 유용한 시각 특징 만들게"
```

> 직관: 답을 틀리면 **"받아쓰는 손"뿐 아니라 "보는 눈"까지** 같이 고쳐진다. 그래서 인코더는 점점 "치킨 글자 영역", "가격 숫자 영역" 같은 **유용한 특징**을 뽑게 된다.

## 4. 무엇이 학습되나 (3가지)

| 학습되는 것 | 의미 |
|---|---|
| **① JSON 구조 문법** | `<s_menu>` 다음엔 보통 `<s_nm>` … 같은 **토큰 순서 규칙**. (이미지 없이도 구조는 어느 정도 앎) |
| **② 글자 읽기 (OCR)** | 특정 시각 패턴(픽셀 모양) → 특정 글자/숫자 토큰 |
| **③ 위치-값 정렬(alignment)** | cross-attention이 **"지금 쓸 토큰은 이미지 어디를 봐야 하는지"** 를 *위치 라벨 없이* 스스로 학습 → OCR의 "어디를 읽을지"를 대신함 = **OCR-free의 핵심** |

---

## 5. 왜 새 영수증에도 되나 (일반화)

학습이 **재사용 가능한 조각**을 익히기 때문:
- 숫자 글리프 `0~9`의 생김새, `Ø`·`±` 같은 기호,
- "`price` 태그 옆엔 숫자가 온다" 같은 레이아웃·문법 패턴,
- "표 윗줄=메뉴, 아랫줄=합계" 같은 구조 관습.

→ 처음 보는 영수증이라도 이 조각들의 조합으로 적절한 토큰열을 **생성**한다.

---

## 6. 학습 ↔ 추론의 결정적 차이 (그리고 함정) ⚠️

| | 학습 (Teacher Forcing) | 추론 (자기회귀) |
|---|---|---|
| 다음 토큰 예측 시 입력 | **정답** 이전 토큰 | **자기가 만든** 이전 토큰 |
| 한 칸 틀리면 | 다음 칸은 정답 줄로 복구 | **오류가 누적**(앞 글자 틀리면 뒤도 흔들림) |

이 불일치를 **노출 편향(exposure bias)** 이라 한다. 데이터가 충분하면 추론도 안정적이지만,
**데이터가 적으면** 학습 손실은 낮은데 추론은 무너지는 현상(첫 글자만 읽고 멈춤, 반복)이 생긴다.

> 📌 이 프로젝트의 **요소 실험에서 본 "0% 사용불가"** 가 바로 이것이다 — 자기 학습 데이터조차 추론으론 못 읽음(데이터 ~1,975장 한계).
> 자세히는 [`Element_Donut_성능미달_원인_및_해결방안.md`](yolo_finetune_donut_pipeline/Element_Donut_성능미달_원인_및_해결방안.md).

---

## 7. 특수 토큰 등록이 학습에 필요한 이유

`<s_menu>` 같은 필드 토큰이 **vocab에 없으면 "정답 토큰"으로 존재할 수 없어** 채점·생성 대상이 안 된다.
그래서 학습 전 **등록 + 임베딩 리사이즈**가 필수. 새로 생긴 임베딩 행은 **랜덤값**에서 시작해,
위 채점·교정 반복으로 **"이 토큰의 의미"** 를 점점 획득한다. (절차는 파인튜닝 가이드 §3)

## 부록 A. 추론 흐름 (학습된 매핑을 쓰는 쪽)

학습으로 만든 매핑을 추론 때 그대로 사용한다 — **이미지 1장 + `<s_cord-v2>` → JSON**:

```
영수증 이미지 ─①전처리(1280×960·정규화)→ pixel_values ─②인코더(Swin)→ 시각 특징 시퀀스
                                                                        │ (cross-attention 참조)
"<s_cord-v2>" ─③토큰화→ 디코더 시작 입력 ─④디코더 자기회귀 생성(</s>까지)→ 토큰열
   ─⑤batch_decode→ 문자열 ─⑥특수·task 토큰 제거 ─⑦token2json→ { "menu":{…}, "total":{…} }
```

- **인코더 = 눈**(이미지→특징), **디코더 = 받아쓰는 손**(특징 보며 토큰 한 개씩), **`<s_cord-v2>` = 모드 스위치**.
- 학습 때와 유일한 차이: ④에서 입력이 "정답 이전 토큰"이 아니라 **자기가 만든 이전 토큰**(→ §6 노출 편향).

---

## 한 문장 요약

> **수천 장의 (이미지, 정답토큰열) 쌍에 대해, 정답을 보여주며 "다음 한 토큰"을 맞히게 시키고(Teacher Forcing),
> 틀린 만큼 디코더와 (cross-attention 통해) 인코더까지 미세 교정하는 반복** — 이것이 "이미지 → JSON 토큰열" 매핑을 만든다.
> cross-attention이 위치 라벨 없이 "어디를 읽을지"를 익히는 게 **OCR-free의 비밀**이다.
